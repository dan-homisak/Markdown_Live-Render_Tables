# Repository Instructions

- Before finishing any code change, run the project type check or compile command and resolve all reported errors.
- After every prompt that changes this repository, run `./Build_and_Install` before responding so the latest extension build is installed into VS Code.
- When making, troubleshooting, or reviewing changes that can affect the user interface, inspect the rendered content directly before finishing. Use the available local harness, screenshots, Extension Development Host, or another direct visual review path; do not rely only on terminal output or user-provided screenshots.
- Do not leave known red-highlight VS Code Problems diagnostics in touched files. If a diagnostic cannot be reproduced from the terminal, inspect and resolve it in the editor context before considering the change complete.

## Live Editor Visual-Parity Notes

- The standalone HTML/Chrome harness is useful only for catching obvious webview regressions. It is not a valid proof that the live editor matches the stock VS Code editor, because it does not include Monaco, VS Code workbench layout, actual editor settings, minimap, sidebar/chat layout, or webview embedding behavior.
- For VS Code visual-parity work, compare stock and live editors inside the same isolated VS Code/Electron window. Launch an Extension Development Host with `--remote-debugging-port`, open `TestTable.md`, capture the stock editor, run `Markdown Live Editor: Toggle Markdown Live Editor`, then capture the live editor in the same workbench layout.
- Use the Electron DevTools endpoint at `http://127.0.0.1:<port>/json` and connect with a local Node script using `undici.WebSocket`. Capture screenshots with `Page.captureScreenshot` and inspect DOM/computed metrics with `Runtime.evaluate`. This avoids the Chrome crash-report noise seen when driving separate headless Chrome windows.
- Dismiss first-run VS Code onboarding/sign-in modals in isolated profiles before judging screenshots. Verify the live editor is actually active by checking that the workbench contains a webview `iframe`; verify the stock editor is active by checking for Monaco `.view-lines`.
- Do not claim visual parity from terminal output or the harness alone. Save and inspect same-layout stock/live screenshots, and compare computed font family, font size, line height, line-number gutter position, first-line x/y, and active-line background.
- For pixel-perfect checks, run `npm run compile`, then run `node scripts/edh-visual-check.mjs` directly. The script opens an isolated Extension Development Host, saves `qa/edh-stock.png` and `qa/edh-live.png`, and fails if core Monaco-vs-live measurements drift by more than 0.5 px. Treat a passing script as necessary but still inspect the screenshots when changing layout.
- Pixel checks must compare actual glyph/text-node boxes as well as element boxes. CodeMirror line boxes can align while text ink is still shifted by default padding; compare Monaco first-line text, line-number text, content-left, gutter width, font size, line height, table-cell left, and active gutter background.
- When diagnosing gutter or text drift, account for coordinate spaces: Monaco measurements are workbench-screen coordinates, while webview measurements are relative to the webview document. Add the Monaco editor-left offset to live webview boxes before comparing x positions.
- Prior visual issue to watch for: the live editor once looked slightly zoomed compared with stock VS Code because it used unzoomed `editor.fontSize`/derived line-height values from the extension host. Prefer VS Code's injected webview CSS variables, especially `--vscode-editor-font-size`, for rendered editor metrics. The line-number gutter was also too far left; preserve Monaco-like gutter geometry by comparing Monaco margin/content rectangles against CodeMirror `.cm-gutters`, `.cm-lineNumbers .cm-gutterElement`, `.cm-content`, and `.cm-line` in the same VS Code window before changing styling.
- Avoid rendering table source line numbers as a separate CodeMirror gutter widget synchronized by `ResizeObserver`. That model is fragile when table rows wrap at different window widths. Table row line numbers should be owned by the rendered table DOM itself so row height and line-number height are resolved by a single layout. If table line numbers disappear, check for clipping and stacking first: the table intentionally shifts left into the gutter, so its wrapper must allow visible overflow and sit above the CodeMirror gutter layer.
